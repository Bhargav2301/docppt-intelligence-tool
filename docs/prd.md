# Product Requirements Document: DocPPT Intelligence Tool

Generated for the Google Docs summarization and PowerPoint humanization product. This PRD defines the product purpose, users, core capabilities, constraints, success metrics, and release boundaries for an MVP that avoids paid Claude/OpenAI-style API dependency by using free, local, and small-model NLP pipelines.

## App Identity

| Field | Value |
| :--- | :--- |
| App Name | DocPPT Intelligence Tool |
| Short Name | DocPPT Intel |
| Tagline | Convert client docs into product specs and clean AI-looking slide text into presentation-ready language. |
| Product Category | Document intelligence, product-spec generation, presentation text QA |

## Problem Statement

Product consultants, founders, PMs, technical writers, and freelance builders often receive client requirements in messy Google Docs and AI-assisted PowerPoint decks. Google Docs may contain long, vague, or unstructured product descriptions that must be converted into concise product descriptions and implementation requirements before development can begin. PowerPoint decks may contain citation artifacts, delimiter issues, special characters, repetitive AI-style wording, uniform rhythm, and overly polished language that makes the content feel machine-generated or unreviewed.

The current manual workflow is slow, inconsistent, and expensive when routed through paid LLM APIs. Users need a local or low-cost tool that can analyze Google Docs, generate structured product artifacts, inspect PPT text quality, flag AI-like writing signals, preserve slide meaning, and export a cleaned presentation without recurring API charges.

## Target Users

### Primary Persona

- **Product development consultant**: Receives rough client docs and needs to convert them into usable product descriptions, requirements, and implementation handoff notes quickly.
- **Technical content architect**: Reviews decks and project briefs before they are shown to clients or development teams.
- **Indie hacker or solo founder**: Wants structured product documentation without paying for repeated LLM API calls.

### Secondary Persona

- **Student or job seeker building portfolio projects**: Needs professional project documents and cleaned PPT deliverables.
- **Agency operations lead**: Standardizes document-processing quality across repeated client intake workflows.

## Goals

- **Reduce doc-to-spec time**: Convert a Google Doc into summary, product description, and implementation requirements in under three minutes for typical documents.
- **Remove recurring API costs**: Run summarization, analysis, and humanization locally or with free/open-source models by default.
- **Improve PPT professionalism**: Detect and rewrite presentation text that contains citation artifacts, awkward delimiters, special characters, repeated AI-style phrases, and overly uniform sentence patterns.
- **Preserve user intent**: Ensure generated summaries, requirements, and rewritten PPT text preserve the source meaning and do not invent features.
- **Create implementation-ready outputs**: Produce outputs that can be copied into PRDs, TRDs, implementation plans, and development tickets.

## Non-Goals

- **Perfect AI detection**: The product must not claim it can prove whether text was generated by AI. AI-likeness scores are probabilistic quality signals.
- **Academic misconduct or deception**: The product should improve clarity, remove artifacts, and make writing natural. It must not market itself as a tool for bypassing academic or compliance detection.
- **Full design automation**: The MVP does not redesign slide layouts, create new visual themes, or generate new slide decks from scratch.
- **Enterprise document governance**: The MVP does not include advanced DLP, legal hold, eDiscovery, or enterprise retention workflows.
- **Multi-language support**: English-only processing for MVP unless the user explicitly enables experimental multilingual models later.

## Product Scope

### Google Docs Processing

The Google Docs workflow accepts a Google Doc link, exported `.docx`, or pasted document text. The app extracts content, cleans formatting noise, segments the document, summarizes the client request, identifies product intent, and generates implementation requirements.

### PowerPoint Processing

The PowerPoint workflow accepts `.pptx` files. The app extracts text by slide and shape, checks for citation artifacts, delimiters, special characters, repetitive phrasing, low burstiness, low lexical variety, and other AI-like patterns, then creates a cleaned version of the deck while preserving slide layout and non-text elements.

### Output Management

Each processing run creates a session with source metadata, analysis results, rewritten outputs, downloadable files, and optional user edits.

### Should Have

- **Session history**: Store recent doc and PPT processing sessions per user.
- **Markdown export**: Download Google Doc-derived outputs as `.md`.
- **JSON export**: Download structured requirements as JSON for downstream automation.
- **Sensitivity presets**: Provide conservative, balanced, and aggressive PPT cleanup modes.
- **Model fallback handling**: If the abstractive small model is unavailable, fall back to extractive summarization and rule-based requirement extraction.

### Could Have

- **Batch processing**: Process multiple docs or PPTs in one run.
- **Custom tone profile**: Choose executive, consultant, technical, or conversational rewrite style.
- **Source traceability viewer**: Link each requirement to the source paragraph or slide.
- **Local desktop package**: Package with Docker or Electron for fully offline usage.
- **Optional cloud GPU mode**: Allow user-provided Hugging Face endpoint or self-hosted inference server, but keep local CPU mode as default.

## User Stories

- **As a consultant**, I want to paste a Google Doc link so that I can generate a product description and implementation requirements without manually reading the entire brief.
- **As a PM**, I want extracted implementation requirements grouped by function, data, integrations, UI, and non-functional constraints so that I can hand them to engineering.
- **As a technical writer**, I want a slide-by-slide list of citation and delimiter artifacts so that I can clean a deck before client delivery.
- **As a founder**, I want the app to use free or local models so that I can process documents without ongoing API charges.

## Functional Requirements

### Google Docs Workflow

- Accept Google Doc URLs, uploaded `.docx`, and pasted text.
- Extract document title, headings, paragraphs, tables where feasible, bullets, and ordered lists.
- Normalize formatting artifacts and preserve heading hierarchy.
- Run summary generation using a hybrid pipeline:
  - Extractive summarization with TextRank, LexRank, or centroid-based ranking.
  - Optional local abstractive summarization using small models such as `sshleifer/distilbart-cnn-12-6`, `google/flan-t5-small`, `t5-small`, or equivalent SLM.
  - Chunking and merge strategy for long documents.
- Generate product description with:
  - Product name or inferred placeholder.
  - Target users.
  - Problem.
  - Proposed solution.
  - Key benefits.
  - Primary workflow.
  - Constraints and assumptions.
- Generate implementation requirements grouped into:
  - Functional requirements.
  - Technical requirements.
  - Integrations.
  - Data requirements.
  - UI/UX requirements.
  - Security and privacy requirements.
  - Performance and reliability requirements.
  - Open questions.

### PPT Humanization Workflow

- Accept `.pptx` files and reject unsupported formats.
- Extract text by slide, shape, paragraph, and run where practical.
- Preserve original deck structure and non-text elements.
- Run artifact detection before rewrite:
  - Citation markers.
  - Markdown remnants.
  - Unsupported delimiters.
  - Unicode inconsistencies.
  - Excessive punctuation.
  - Inconsistent bullet nesting.
- Run AI-likeness quality analysis:
  - Tokenization and sentence segmentation.
  - Perplexity estimation using local reference LM.
  - Burstiness analysis across sentence length and local perplexity.
  - Lexical diversity and type-token ratio.
  - N-gram repetition.
  - POS pattern or simple syntax feature extraction when spaCy is available.
  - Punctuation density and connective phrase detection.
  - Optional classical ML or LSTM classifier trained on user-approved corpora.
- Rewrite only segments that need revision.
- Preserve factual meaning and slide intent.
- Offer review controls before export.

## Non-Functional Requirements

- **Cost**: MVP must not require paid LLM APIs. All NLP processing should run locally or through free/open-source models.
- **Performance**: Typical Google Doc processing should complete within three minutes on a standard CPU environment for MVP-sized inputs.
- **Reliability**: If a model fails, the app should gracefully fall back to deterministic extraction and rule-based analysis.
- **Privacy**: User documents must not be sent to paid external LLM APIs by default.
- **Explainability**: PPT flags should show why text was flagged, not just output a score.
- **User control**: Rewrites should be reviewable and reversible.
- **Export fidelity**: PPT output must preserve slide count, order, layout, images, charts, and shapes.
- **Accessibility**: UI must support keyboard navigation, readable contrast, and screen-reader-friendly processing states.

## Success Metrics

| Metric | Target |
| :--- | :--- |
| Google Doc processing completion rate | At least 90% for valid supported inputs |
| PPT export fidelity | 100% slide count preservation and no known layout-breaking changes |
| Local/free model usage | 100% of MVP NLP tasks completed without paid Claude/OpenAI-style API calls |
| Summary usefulness | At least 80% of test users rate summaries as useful or very useful |
| Requirement extraction usefulness | At least 75% of extracted requirements accepted without major rewrite in test review |
| PPT cleanup acceptance | At least 80% of rewritten segments accepted or lightly edited by users |
| False positive management | Users can dismiss or revert 100% of questionable PPT flags before export |

## Risks and Mitigations

| Risk | Impact | Mitigation |
| :--- | :--- | :--- |
| Small models generate weak summaries | Output quality may be inconsistent | Use hybrid extractive plus abstractive pipeline and source-grounded templates. |
| Local CPU inference is slow | Poor user experience | Chunk input, cache models, add progress states, and provide extractive fallback. |
| AI-likeness scoring is unreliable | Users may overtrust scores | Label scores as quality signals, show explanations, and require review before export. |
| PPT formatting breaks | User loses trust | Modify text runs conservatively and preserve original slide structure. |
| Long Google Docs exceed model context | Missing content in summary | Chunk by heading, summarize chunks, then synthesize final output. |
| Rewrites alter meaning | Client requirements may be distorted | Add semantic similarity checks and user review before applying changes. |

## Open Questions

- Should Google Docs integration be OAuth-based from day one, or should MVP start with `.docx` upload and pasted text?
- What is the maximum expected PPT size and slide count?
- Should users be able to train or calibrate the LSTM/classical detector on their own writing samples?
